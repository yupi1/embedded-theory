## Що це за проєкт

**Embedded Notes** — статичний сайт на GitHub Pages з технічними конспектами для підготовки до інтерв'ю. Замінює Notion: тут є SVG-діаграми, анімації, інтерактивний Q&A.

Стек: чистий HTML + CSS + JS. Без фреймворків. Кожна сторінка — самодостатній `.html` файл.

**Дизайн-система:** темна тема з осцилографною естетикою. JetBrains Mono + IBM Plex Sans. Акцент `#00d084` — зелений термінального типу.

## Акцентний колір — логіка вибору

Колір передається через `--topic-color` у `:root` кожної сторінки.
Вибір базується на семантиці сигналу — як на осцилографі:

| Категорія        | `--topic-color`     | Hex       | Логіка                          |
|------------------|---------------------|-----------|---------------------------------|
| Електроніка      | `var(--rising)`     | `#00d084` | Зелений = стабільний сигнал     |
| Протоколи        | `var(--clk)`        | `#4a9eff` | Синій = тактовий сигнал CLK     |
| Мікроконтролер   | `var(--falling)`    | `#ff6b4a` | Помаранчевий = переривання      |
| Пам'ять          | `var(--warning)`    | `#f5c842` | Жовтий = запис/читання          |
| Архітектура      | `var(--clk)`        | `#4a9eff` | Синій = системний рівень        |
| RTOS             | `var(--warning)`    | `#f5c842` | Жовтий = часові аспекти         |

## Структура теми — еталонний порядок секцій

Кожна тема — це навчальний конспект що будує розуміння крок за кроком, а не просто reference card.

### Обов'язкові секції (у цьому порядку):

**01 — Визначення**
Що це? Одне чітке визначення + аналогія. Мінімум один `callout-important` з головною ідеєю.

**02 — Лінії зв'язку / Компоненти** *(для протоколів)*
Список ліній або компонентів з поясненням кожного. `.line-item` або таблиця.

**03 — Налаштування / Принцип роботи**
Кроки налаштування (`.steps-list`) або принцип роботи. Для числових параметрів — формула у code-block.

**04 — Data Frame / Структура** *(якщо є)*
SVG тайм-діаграма або схема всередині `.waveform-demo`. Таблиця з описом кожної частини.

**05 — Принцип роботи** *(детально)*
Якщо тема складна — окрема секція для deep-dive. SVG-анімація механізму.

**06 — Відстань / Характеристики** *(якщо є числа)*
`.stat-grid` зі stat-картками.

**07 — Будова / Компоненти модуля**
Таблиця або список компонентів апаратного модуля.

**08 — Переваги / Недоліки**
Два `.topic-list` — один для переваг, один для недоліків.

**09 — Код** *(обов'язково!)*
`.code-block` з HAL або ESP-IDF прикладом. Підсвічування через span-класи.

**10+ — Питання на співбесіді** *(обов'язково!)*
`.accordion` мінімум 4–6 питань. Кожна відповідь — повна і детальна.

### Опційні секції:
- Правила/обмеження (для переривань, RTOS)
- Порівняльна таблиця (SPI vs I²C, polling vs interrupt)
- Приклад розрахунків (формули з поясненнями)

## Компоненти і коли їх використовувати

**`.ic`** — inline-code для назв реєстрів, функцій, значень у тексті. Наприклад: регістр `<code class="ic">ARR</code>`.

**`.badge .badge-*`** — теги статусу в шапці (`badge-rising` = learnt, `badge-neutral` = soon/platform).

**`.callout .callout-*`** — виділені блоки:
- `callout-important` + `KEY` → головна ідея секції
- `callout-info` + `INF` → технічна деталь або порада
- `callout-warning` + `WRN` → типова помилка або застереження
- `callout-danger` + `ERR` → критична небезпека (наприклад, спалити MCU)

**`.waveform-demo`** — контейнер для будь-яких SVG-діаграм і тайм-діаграм.

**`.kit-table` у `.table-wrap`** — таблиці визначень, порівнянь, регістрів.

**`.stat-grid` + `.stat-card`** — числові параметри: частота, швидкість, напруга.

**`.code-block`** — блок коду з header (назва + copy кнопка) і body. Підсвічення:
- `span.kw` → синій (ключові слова, типи)
- `span.fn` → зелений (функції)
- `span.reg` → жовтий (регістри)
- `span.num` → помаранчевий (числа)
- `span.cm` → сірий курсив (коментарі)
- `span.str` → жовтий (рядки)

**`.accordion`** — Q&A блок. Один `accordion-item` = одне питання.

**`.topic-list`** — ненумерований список з тире (`—`) як маркером.

**`.steps-list`** — нумерований список з кружечками.

**`.def-cards`** — 3 картки в ряд для визначень термінів (як Interrupt / ISR / Vector Table).

## Як додати нову тему — покрокова інструкція

### Крок 1 — Створити HTML файл
```bash
cp TEMPLATE.html protocols/spi.html
```

### Крок 2 — Заповнити заголовок у файлі
- `<title>SPI · Конспект · Embedded Systems</title>`
- `--topic-color: var(--clk);` у `:root`
- `page-eyebrow`: `Конспект · Protocols`
- `page-title` з акцентним словом
- `page-desc`: 1–3 речення опис
- `page-tags`: badge статусу + платформи

### Крок 3 — Заповнити секції за еталонним порядком
Дивись "Структура теми" вище. Не пропускай обов'язкові секції.

### Крок 4 — Додати тему в index.html
Відкрий `index.html`, знайди масив `TOPICS` і додай об'єкт:

```javascript
{
  category: 'protocols',        // папка
  catLabel: 'Протоколи',        // назва категорії (UA)
  catColor: '#4a9eff',          // колір категорії
  catBadge: 'badge-clk',        // клас badge для категорії
  path:   'protocols/spi.html', // шлях до файлу
  title:  'SPI',                // назва (коротко)
  accent: 'Serial Peripheral Interface', // підзаголовок
  status: 'done',               // 'done' або 'soon'
},
```

Змінити `status: 'soon'` на `status: 'done'` — картка стане активною.

## Правила написання контенту

**Мова:** основний текст — українська. Назви реєстрів, функцій, протоколів — англійська, завжди у `<code class="ic">`.

**Стиль:** навчальний конспект, а не reference. Кожна секція будує розуміння на попередній. Пояснюй "чому", а не тільки "що".

**Код:** завжди `data-lang` або текст у `.code-block-lang`. Підсвічуй токени вручну. Коментарі в коді — українська.

**Q&A:** відповіді мають бути повними і детальними. Якщо питання складне — кілька абзаців з `<p>` і `<p class="mt-6">`.

**Критерій "готова сторінка":** є визначення + callout-important, таблиця або список ключових компонентів, хоча б один code-block, мінімум 4 Q&A у accordion, картка в index.html зі статусом `done`.

## Як публікувати на GitHub Pages

1. Репозиторій має назву `username.github.io` або будь-яка назва.
2. Settings → Pages → Source: `main` branch, папка `/` (root).
3. Commit і push — сайт буде на `https://username.github.io/embedded-notes/`.
4. Нові сторінки: просто додай файл у правильну папку і оновлюй `TOPICS[]` в `index.html`.

## Спільний доступ для друга

**Варіант 1 — Collaborator (рекомендовано для двох):**
GitHub → Settings → Collaborators → Add people → email або username. Друг зможе робити push напряму.

**Варіант 2 — Fork + Pull Request:**
Друг робить fork репозиторію, додає зміни у своєму форку, відкриває Pull Request. Ти ревʼюєш і mergeуєш. Підходить якщо хочеш контролювати що потрапляє на сайт.

**Варіант 3 — Branch-based:**
Кожен працює у своїй гілці (`anton/uart`, `friend/i2c`). Потім merge в `main`. Найгнучкіший варіант.

Для embedded notes — варіант 1 (Collaborator) найпростіший.
